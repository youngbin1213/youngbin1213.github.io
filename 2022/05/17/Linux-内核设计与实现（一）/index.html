进程管理

对 Linux而言，线程不过是一种特殊的进程

进程是处于执行期的程序以及相关资源的总称

进程提供两种虚拟机制——虚拟处理器和虚拟内存，也就是说进程和进程之间的相互隔离

通常通过调用fork产生进程。调用fork()结束时，返回到相同的位置，父进程恢复执行，子进程开始执行。fork()系统调用从内核返回两次：一次返回到父进程，一次返回到子进程

程序通过exit()系统调用退出执行，这个函数会将占用的资源释放。父进程可以通过wait4()来查询子进程是否终结，可以用来等待特定进程。进程退出执行后被设置为僵死状态，直到它的父进程调用wait()或 waitpid()

task(任务)是进程的另一个名字，后续回穿插使用

进程描述符及任务结构

内核把进程的列表放在 任务队列里(task list) 双向循环链表，链表每一项都是 类型为task_struct，称为进程描述符（process descriptor）的结构，该结构定义在 <linux/sched.h>中。

task_struct 在32位机器中，大约有1.7KB

使用 slab分配器分配 task_struct 结构 。能达到对象复用和缓存着色（cache coloring)(后面会提到)(不懂)

task_struct 存放在内核栈尾端

使用slab分配器动态生成task_struct，所以只需要在栈底（对于乡下生长的栈）或者栈顶（向上生长的栈）创建一个 struct thread_info(为什么不是新建一个 task_struct？？？)

struct thread_info {
	struct task_struct	*task;		/* main task structure */
	struct exec_domain	*exec_domain;	/* execution domain */
	unsigned long		flags;		/* low level flags */
	unsigned long		status;		/* thread-synchronous flags */
	__u32			cpu;		/* current CPU */
	__s32			preempt_count; /* 0 => preemptable, <0 => BUG */


	mm_segment_t		addr_limit;	/* thread address space:
					 	   0-0xBFFFFFFF for user-thead
						   0-0xFFFFFFFF for kernel-thread
						*/
	struct restart_block    restart_block;

	unsigned long           previous_esp;   /* ESP of the previous stack in case
						   of nested (IRQ) stacks
						*/
	__u8			supervisor_stack[0];
};

进程描述符的存放

内核通过唯一一个的进程标识值（process identification value)PID 来识别一个数

为了兼容旧版Linux和Unix，PID的最大值默认位 32768

内核中访问任务（进程），通常需要 它的task_struct指针。基本上都提供硬件上的加速。有的硬件体系结构可以拿出一个专门的寄存器来存放 task_struct指针。x86(寄存器并不富裕)。可以在尾端创建 thread_info结构，并且通过偏移查找到 task_struct 结构

进程上下文

当一个程序执行了系统调用（参见第五章）或者触发了某个异常，它就陷入了内核空间。此时，内核”代表进程执行“并处于进程上下文中。

系统调用和异常处理程序是对内核明确定义的接口。进程通过这些接口才能陷入内核执行

进程创建

许多其它操作系统都提供产生（spawn）进程的机制。

Unix采用与众不同的实现方式，将上述步骤分解到两个单独的函数执行: fork 和 exec 。

  • fork :通过拷贝当前进程创建子进程 区别 （PID、PPID、某些资源和统计量（挂起的信号））
  • exec:读取可执行文件并将其载入地址空间开始运行

写时拷贝（copy-on-write)

结论： 可以推迟甚至免除拷贝数据的技术

实现方式：只有在写入时，数据才会被复制，在此之前，只是通过已读的方式共享

fork 的唯一开销是复制父进程的页表以及给子进程创建唯一的进程描述符。这种优化可以避免拷贝大量根本就不会被使用的数据（地址空间中常常包含数十兆的数据）

fork函数

asmlinkage int sys_fork(struct pt_regs *regs)
{
	return do_fork(SIGCHLD, regs->ARM_sp, regs, 0, NULL, NULL);
}

asmlinkage int sys_clone(unsigned long clone_flags, unsigned long newsp,
			 int __user *parent_tidptr, int tls_val,
			 int __user *child_tidptr, struct pt_regs *regs)
{
	if (!newsp)
		newsp = regs->ARM_sp;

	return do_fork(clone_flags, newsp, regs, 0, parent_tidptr, child_tidptr);
}

asmlinkage int sys_vfork(struct pt_regs *regs)
{
	return do_fork(CLONE_VFORK | CLONE_VM | SIGCHLD, regs->ARM_sp, regs, 0, NULL, NULL);
}

do_fork 完成了创建中的大部分工作，定义在 kernel/fork.c中。该函数调用 copy_process()函数，然后让进程开始运行。

static task_t *copy_process(unsigned long clone_flags,
				 unsigned long stack_start,
				 struct pt_regs *regs,
				 unsigned long stack_size,
				 int __user *parent_tidptr,
				 int __user *child_tidptr,
				 int pid)
{
    ... // 检测
    p = dup_task_struct(current);// 创建 子 task_struct
    retval = -EAGAIN;
    // 检测当前用户拥有的进程数量没有超过给它分配的资源限制
    // 第二个if 使用了 capable() 函数来对权限做出检查，检查是否有权限 ，无权限 返回 0
    if (atomic_read(&p->user->processes) >=
			p->signal->rlim[RLIMIT_NPROC].rlim_cur) {
		if (!capable(CAP_SYS_ADMIN) && !capable(CAP_SYS_RESOURCE) &&
				p->user != &root_user)
			goto bad_fork_free;
	}
    //增加用户进程数
    atomic_inc(&p->user->__count);
	atomic_inc(&p->user->processes);
    //将一些值设为0（主要是统计值）
    p->did_exec = 0;
    p->rchar = 0;		/* I/O counter: bytes read */
	p->wchar = 0;		/* I/O counter: bytes written */
	p->syscr = 0;		/* I/O counter: read syscalls */
	p->syscw = 0;		/* I/O counter: write syscalls */
    // 确保子进程不会投入使用
    p->did_exec = 0;
    // 设置pid
    copy_flags(clone_flags, p);

    p->pid = pid;
    /* copy all the process information */
	if ((retval = copy_semundo(clone_flags, p)))
		goto bad_fork_cleanup_audit;
	if ((retval = copy_files(clone_flags, p)))
		goto bad_fork_cleanup_semundo;
	if ((retval = copy_fs(clone_flags, p)))
		goto bad_fork_cleanup_files;
	if ((retval = copy_sighand(clone_flags, p)))
		goto bad_fork_cleanup_fs;
	if ((retval = copy_signal(clone_flags, p)))
		goto bad_fork_cleanup_sighand;
	if ((retval = copy_mm(clone_flags, p)))
		goto bad_fork_cleanup_signal;
	if ((retval = copy_keys(clone_flags, p)))
		goto bad_fork_cleanup_mm;
	if ((retval = copy_namespace(clone_flags, p)))
		goto bad_fork_cleanup_keys;


}

copy_process函数步骤：

1. 调用 dup_task_struct() 创建内核栈、thread_info结构和task_struct。此时 **子进程和父进程的描述符**是完全相同的
1. 检查当前用户拥有的进程数量没有超过给它分配的资源限制
1. 将一些值设为0（主要是统计值）
1. 子进程的did_exec 被设为 0 ，以确保它不会投入运行
1. 设置flags
1. 设置pid
1. 根据 `clone_flags`参数信息拷贝文件、文件系统信息
1. 返回一个指向子进程的指针

线程在Linux中的实现

线程机制提供了在同一程序内共享内存地址空间运行。线程机制支持并发程序设计(concurrent programming)，在多处理器系统上，也能保证真正的并行处理（parallelism）。

Linux实现线程的机制很独特。从内核的角度看，Linux并没有线程这个概念。把所有的线程都当作进程来实现。内核并没有准备特别的调度算法或是定义特别的数据结构来表征线程。线程被看作一个与其它进程共享某些资源的进程。每个线程都拥有唯一隶属于自己的task_struct。所以在内核中，它看起来就像一个普通的进程

线程的创建

调用clone 函数创建线程

// create thread

参数含义

内核线程

内核线程用于在后台执行一些操作。内核线程（kernel thread)是独立运行在内核空间的标准进程（linux的线程和进程）。

内核线程和普通进程的区别：内核线程没有独立的地址空间 （mm指针被设为 NULL），只在内核空间运行，不会被切换到用户空间。（被称为线程的原因就是没有独立的地址空间）

相同点：可以被调度和中断

Linux 会把一些任务交给内核线程 -> 比如 flush和 ksofirqd

内核线程只能由其它内核线程创建，在 linux/kthread.h中申明有接口

struct task_struct *kthread_create(int (*threadfn)(void *data),
				   void *data,
				   const char namefmt[], ...);

新的任务是由kthread内核通过 clone()系统调用。新的进程将运行 threadfn 函数，函数传递的参数为 data。进程会被命名为 namefmt.新创建的进程处于不可运行状态，需要通过 wake_up_process 让它运行

/**
 * kthread_run: create and wake a thread.
 * @threadfn: the function to run until signal_pending(current).
 * @data: data ptr for @threadfn.
 * @namefmt: printf-style name for the thread.
 *
 * Description: Convenient wrapper for kthread_create() followed by
 * wake_up_process().  Returns the kthread, or ERR_PTR(-ENOMEM). */
#define kthread_run(threadfn, data, namefmt, ...)			   \
({									   \
	struct task_struct *__k						   \
		= kthread_create(threadfn, data, namefmt, ## __VA_ARGS__); \
	if (!IS_ERR(__k))						   \
		wake_up_process(__k);					   \
	__k;								   \
})

内核线程启动后一直运行到 调用do_exit()退出，或者内核其它部分调用 kthread_stop()

进程终结

进程总归是要终结的，内核需要释放它占用的所有资源，并把这一不幸告知其父进程

不管主动终结还是被动终结，大部分是依靠 do_exit() (位于 kernel/exit.c)

fastcall NORET_TYPE void do_exit(long code)
{
	struct task_struct *tsk = current;
	int group_dead;
	
	profile_task_exit(tsk);

	if (unlikely(in_interrupt()))
		panic("Aiee, killing interrupt handler!");
	if (unlikely(!tsk->pid))
		panic("Attempted to kill the idle task!");
	if (unlikely(tsk->pid == 1))
		panic("Attempted to kill init!");
	if (tsk->io_context)
		exit_io_context();

	if (unlikely(current->ptrace & PT_TRACE_EXIT)) {
		current->ptrace_message = code;
		ptrace_notify((PTRACE_EVENT_EXIT << 8) | SIGTRAP);
	}
	// set FLAG PF_EXITING
	tsk->flags |= PF_EXITING;
    // 删除任一内核定时器。
	del_timer_sync(&tsk->real_timer);

	if (unlikely(in_atomic()))
		printk(KERN_INFO "note: %s[%d] exited with preempt_count %d\n",
				current->comm, current->pid,
				preempt_count());
	// 输出记账信息？？？？？？？ 没搞懂什么意思 
	acct_update_integrals();
	update_mem_hiwater();
	group_dead = atomic_dec_and_test(&tsk->signal->live);
	if (group_dead)
		acct_process(code);
    // 释放 进程占用的 mm_struct 如果没用别的进程使用（即地址空间未被共享），彻底释放
	exit_mm(tsk);
	// 如果current 进程在等待 IPC信号，则离开队列
	exit_sem(tsk);
    // 递减 文件描述符、文件系统数据的引用计数。如果降为0，那么就代表没有进程在使用相应的资源，此时应该释放
	__exit_files(tsk);
	__exit_fs(tsk);
	exit_namespace(tsk);
	exit_thread();
	exit_keys(tsk);

	if (group_dead && tsk->signal->leader)
		disassociate_ctty(1);

	module_put(tsk->thread_info->exec_domain->module);
	if (tsk->binfmt)
		module_put(tsk->binfmt->module);

	tsk->exit_code = code;
    //向父进程发送信号，给子进程重新找养父，一般为 init
	exit_notify(tsk);
#ifdef CONFIG_NUMA
	mpol_free(tsk->mempolicy);
	tsk->mempolicy = NULL;
#endif

	BUG_ON(!(current->flags & PF_DEAD));
    // 调用 scheduler 切换到新进程
	schedule();
	BUG();
	/* Avoid "noreturn function does return".  */
    /* 处于 EXIT_ZOMBIE 状态 永不返回 */
	for (;;) ;
}

运行 do_exit之后进程不可运行（实际上也没地址空间让它运行）并处于 EXIT_ZOMBIE状态。占用的内存只有 内核栈、thread_info和task_struct结构。进程存在的目的是向父进程提供信息。父进程处理之后，剩下的内存也被释放，归还给系统使用

删除进程描述符

调用 do_exit之后，线程已经僵死无法运行，但是系统仍然保留了进程描述符，这样可以让系统有办法获得它的信息，因此，进程的清理工作和 进程描述符被分开执行。在内核不关注那些信息后，子进程的task_struct 结构才被释放

使用 release_task() 释放进程描述符

void release_task(struct task_struct * p)
{
	int zap_leader;
	task_t *leader;
	struct dentry *proc_dentry;

repeat: 
	atomic_dec(&p->user->processes);
	spin_lock(&p->proc_lock);
	proc_dentry = proc_pid_unhash(p);
	write_lock_irq(&tasklist_lock);
	if (unlikely(p->ptrace))
		__ptrace_unlink(p);
	BUG_ON(!list_empty(&p->ptrace_list) || !list_empty(&p->ptrace_children));
    // 从pidhash和从任务列表中删除该进程
	__exit_signal(p);
	__exit_sighand(p);
	__unhash_process(p);

	/*
	 * If we are the last non-leader member of the thread
	 * group, and the leader is zombie, then notify the
	 * group leader's parent process. (if it wants notification.)
	 */
	zap_leader = 0;
	leader = p->group_leader;
	if (leader != p && thread_group_empty(leader) && leader->exit_state == EXIT_ZOMBIE) {
		BUG_ON(leader->exit_signal == -1);
		do_notify_parent(leader, leader->exit_signal);
		/*
		 * If we were the last child thread and the leader has
		 * exited already, and the leader's parent ignores SIGCHLD,
		 * then we are the one who should release the leader.
		 *
		 * do_notify_parent() will have marked it self-reaping in
		 * that case.
		 */
		zap_leader = (leader->exit_signal == -1);
	}

	sched_exit(p);
	write_unlock_irq(&tasklist_lock);
	spin_unlock(&p->proc_lock);
	proc_pid_flush(proc_dentry);
	release_thread(p);
    // 释放进程内核栈和 thread_info 结构所占的页
	put_task_struct(p);

	p = leader;
	if (unlikely(zap_leader))
		goto repeat;
}
